# LINE Bot (耀西) 運作與架構說明

本文件深入說明 Travel Ledger WebApp 專屬 LINE Bot（耀西）的行為邏輯、架構設計與程式碼說明。
開發者若修改機器人邏輯或自我介紹，請務必同步更新本文件與 `index.ts`。

## 1. 核心行為與功能
- **自然語言記帳**: 透過 Gemini AI 解析使用者的自然語言輸入（例如：「午餐 500 迪哥付的」），自動轉化為結構化的記帳資料。支援追溯性指令，如「剛剛那筆改 500」，AI 會參考對話紀錄進行修正。
- **收據 OCR 智慧辨識**: 
  - **自動流程**: 使用者上傳照片後，機器人會自動下載並同步存儲至 Supabase Storage (`expenses/{tripId}/{messageId}.jpg`)。
  - **AI 解析**: 使用 Gemini 視覺模型辨識商店、金額、幣別、日期與品項，並根據「個人偏好設定」預測分帳比例。
  - **預覽確認**: 解析後會回傳包含收據縮圖的 Flex Message，使用者確認後才正式寫入資料庫；若取消，系統會自動刪除 Storage 中的照片。
- **雙重綁定機制**: 
  - 使用者在 LINE 輸入 `ID:短碼` 啟動綁定流程（進入 `pending_trip_id` 狀態）。
  - 系統要求輸入旅程的 `access_code` 進行驗證，成功後正式綁定 (`current_trip_id`)。
- **群組喚醒邏輯**:
  - 單對單聊天：機器人會回應所有訊息。
  - 群組聊天：必須提及 (Tag) 機器人、觸發「耀西」關鍵字、或執行系統指令（`ID:`, `設定`, `斷開`）才會喚醒。
- **個人偏好設定**: 支援使用者透過「設定：預設付款人是小明」來記憶個人記帳偏好 (`default_config`)，AI 在解析時會以此作為基準。
- **財務完整性校驗**: 
  - **高精度運算**: 內建與前端一致的 `decimal.js` 邏輯處理餘數分配。
  - **最終防線**: 在 `expenses` 表格寫入前，系統會強制校驗 `Σ(付款人金額) == 總額` 且 `Σ(分帳金額) == 總額`。若不符則拒絕寫入。
- **防重複提交 (Idempotency)**: 
  - 每個 Flex Message 確認按鈕皆帶有唯一的 `nonce`。
  - 點擊後會紀錄於 `line_processed_actions` 表，防止因連點或網路延遲產生的重複帳務。
- **封存保護**: 機器人會檢查旅程的 `is_archived` 狀態。若已封存，則僅提供查詢與聊天功能，禁止新增支出。

1. 輸入「ID:您的旅程代碼」

## 3. 技術規格與實作細節 (`index.ts`)
- **狀態管理**: 透過 `line_user_states` 維護使用者的旅程與偏好上下文。
- **對話記憶**: 紀錄 `line_chat_history` (最近 10 筆)，讓 AI 具備脈絡理解能力。
- **Payload 優化**: 由於 LINE Postback Data 有 300 字元限制，程式碼對 JSON 鍵名進行了極端縮寫（例如：`d` 代表 `description`, `a` 代表 `amount`, `cat` 代表 `category` 等）。
- **錯誤處理**: 
  - 針對圖片下載、OCR 解析、資料庫寫入皆有完善的 `try-catch` 保護。
  - OCR 失敗或使用者取消時，會自動清理已上傳的 Storage 暫存檔。
- **多幣別支援**: 根據旅程的 `precision_config` 自動判定金額精度（如 TWD 預設為 0 位小數，其餘為 2 位）。
